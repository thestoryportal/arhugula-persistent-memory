## THE FINDING (K6) — two claims (advisor-refined)
**Claim A — threat-model-INDEPENDENT (solid core).** The spec mandates **no operational-window cryptographic tamper-evidence** for the ledger: keyless Merkle-chain-lite (§16.1; `entry_hash=SHA(prev‖body)`, entries not individually signed — confirmed in G2), integrity = chain-walk from the Genesis Seal (§16.4, internal consistency only), external anchors **only at root** (§13.2 Genesis Seal) **and close** (§16.7/§27 `FINAL_STATE_SNAPSHOT` tip hash). Established by reading §16 end-to-end → elevates CORPUS/11's *prototype* caveat ("not modeled here") to a confirmed *spec-level* property ("not specified"). **Fix:** CT-style Signed Tree Head — periodic offline-signed `(seq,tip)` head-anchor + monotonic seq, verified at boot/cadence; **custody-compatible** with the offline/HSM key model the spec **already mandates for the CAK (§20.2)**. K5 demonstrates it closes K1+K2.

**Claim B — threat-model-SCOPED (sharper; an unreconciled seam, NOT a flat contradiction).** W3 (§16.2) defends against **logical** Orchestrator compromise (it won't mediate W1 honestly → Gate/TC/Boot-verifier *bypass* it to log `INTEGRITY_VIOLATION_EMERGENCY`). Defeating W3 via rewrite-recompute (K1) needs **filesystem-level** tampering — a different capability. The spec's intended ledger-integrity defense for that is **§16.5 access control** (fs-write restricted to Orchestrator/Boot-verifier/Retention-Manager). **Why it doesn't fully cover this:** §16.5 *grants the Orchestrator* ledger fs-write, §16.2 admits the Orchestrator as compromisable, and §5 excludes only **physical** host security — so a *logically*-compromised Orchestrator, running with its normally-granted fs-access, needs no physical/host compromise to rewrite the file. The spec **never reconciles §16.5 with §16.2** → the W3 emergency entry meant to flag the compromise lands in the keyless ledger the same component can rewrite/suppress (K1), with the §16.4 walk still INTACT. **An unreconciled threat-model seam** — live (not hypothetical) precisely because §5's carve-out is physical-only.

## Verdict
Operational-window ledger tamper-evidence is **UNSPECIFIED** (claim A, solid); under §16.2's own threat model it is **security-relevant** because the W3 response shares the mutable substrate with its adversary (claim B, unreconciled seam). **Recommended backstop: STH head-anchor (custody-compatible with §20.2).** Moves **C6** from "verifier-mechanics-proven, not red-teamed" → **"red-teamed: named property-gap (A) + named threat-model seam (B) + named, custody-compatible fix."** A scoped security characterization of ONE mechanism — **NOT a falsification of the spec**, NOT promotable; it **sharpens** the F1 C6 condition (which stays open: prototyped/partially-red-teamed).
