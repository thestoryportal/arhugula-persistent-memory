# MEMIT target model recommendation: a survey of eleven open-weights candidates

**Bottom line up front.** Of the eleven candidates evaluated against the MEMIT-COMPAT-PROTO-1 hard constraints, **seven admit and four reject**. The recommended primary target is **LLaMA 3.1 8B**, with **Mistral 7B v0.3** as the de-risked secondary and **GPT-J 6B** as the canonical-baseline reference for calibrating the editing pipeline before scaling up. These three together cover the full risk spectrum: a modern, well-precedented family-leader (LLaMA 3.1 8B), an Apache-2.0 sweet-spot fallback with identical edit-precedent depth (Mistral 7B v0.3), and the original MEMIT reference model that lets the operator validate the .vindex toolchain against published numbers before committing to a 2024-era target. The four rejected candidates (Phi-3 medium, Falcon 7B, Falcon 40B, Yi 34B) all fail SC-6 — none have a documented MEMIT/ROME precedent paper or first-class EasyEdit support, and Falcon's parallel attention/MLP block additionally raises an SC-5 architectural concern. Layer counts, normalization topologies, and license texts were verified against primary technical papers and config.json files; no aggregator listings were trusted for SC-3 or SC-5.

## 2. Per-candidate evaluation

### LLaMA 3.1 8B
**SC-1** PASS — gated HF repo `meta-llama/Meta-Llama-3.1-8B`, full safetensors plus `original/*` Meta-format weights. **SC-2** PASS — dense decoder-only; the LLaMA 3 herd paper (arXiv:2407.21783) explicitly contrasts the dense Llama-3 family with later MoE variants. **SC-3** PASS — **32 layers**, confirmed in the LLaMA 3 paper Table 3 and in `config.json`'s `num_hidden_layers: 32`. **SC-4** PASS — 128K tiktoken-based BPE (vocab 128,256), shipped in `tokenizer.json` pinnable via HuggingFace revision SHA. **SC-5** PASS — Pre-LN + RMSNorm with SwiGLU and RoPE, identical to LLaMA 1/2; HF config exposes `rms_norm_eps: 1e-05`. **SC-6** PASS (strong) — direct MEMIT precedent in Yoon et al. arXiv:2405.00664 ("Is Bigger Edit Batch Size Always Better?"), AlphaEdit (arXiv:2410.02355, ICLR 2025), Gupta et al. arXiv:2502.01636 (lifelong editing on Llama3-8B with 10K edits), HalluEditBench (arXiv:2410.16251), and EasyEdit's `hparams/MEMIT/llama-7b.yaml` template adapted upstream. **SC-7** PASS — 8.03B parameters. **SC-8** NOTED-DEFERRED. **SC-9** PASS with conditions — LLaMA 3.1 Community License grants modification and derivative-work redistribution under §1.a; §1.b.i imposes a "Built with Llama" attribution, a "Llama-" name prefix on the artifact, and bundling the agreement; §2's 700M-MAU threshold is the only commercial gate and unlikely to bind. **ADMIT.**

### LLaMA 3.1 70B
**SC-1** PASS — gated HF repo. **SC-2** PASS — dense decoder-only. **SC-3** PASS — **80 layers** per LLaMA 3 paper Table 3 and NeMo `Llama31Config70B(num_layers=80)`. **SC-4** PASS — same 128K tiktoken BPE. **SC-5** PASS — Pre-LN + RMSNorm. **SC-6** PASS (borderline) — no LLaMA-3.1-70B-specific MEMIT paper, but Yao et al.'s CaKE (arXiv:2503.16356) explicitly compares against MEMIT on **Llama-3-70B-Instruct** ("CaKE can handle a 70B-sized edit via two A100 GPUs"). EasyEdit's MEMIT YAML applies with parameter-count adjustment. **SC-7** PASS — 70.6B. **SC-8** NOTED-DEFERRED. **SC-9** PASS — same LLaMA 3.1 Community License as 8B. **ADMIT (caveat: SC-6 precedent is at scale, not at the same checkpoint).**

### LLaMA 3.3 70B-Instruct
**SC-1** PASS. **SC-2** PASS. **SC-3** PASS — **80 layers** verified via mirror config.json. **SC-4** PASS — identical tokenizer to 3.1. **SC-5** PASS — same backbone, only post-training changed. **SC-6** PASS (borderline) — no 3.3-specific MEMIT paper exists (release was Dec 2024); precedent inherited from architecturally identical Llama-3.1-70B via CaKE arXiv:2503.16356. **SC-7** PASS — 70B. **SC-8** NOTED-DEFERRED. **SC-9** PASS — LLaMA 3.3 Community License is substantively equivalent to 3.1. **ADMIT (caveat: Instruct-only release; no pretrained base means MEMIT causal-tracing baselines must be re-validated, since the original MEMIT paper localizes facts on base models, not instruction-tuned ones).**

### Mistral 7B v0.3
**SC-1** PASS — `mistralai/Mistral-7B-v0.3` open, ungated. **SC-2** PASS — dense `MistralForCausalLM` (Mixtral 8x7B is the separate MoE variant explicitly excluded). **SC-3** PASS — **32 layers**, `num_hidden_layers: 32` in config.json. **SC-4** PASS — SentencePiece BPE with byte-fallback, vocab extended from 32,000 to 32,768 in v0.3 (`tokenizer.model.v3`); pinnable via HF revision SHA. **SC-5** PASS — Pre-LN + RMSNorm, LLaMA-style. **SC-6** PASS (strong) — RLEdit (arXiv:2502.05759) runs MEMIT/AlphaEdit/RECT/PRUNE on **Mistral-7B-v0.3 by name**; HalluEditBench (arXiv:2410.16251) runs MEMIT/ROME/FT-M on Mistral-v0.3-7B; MEMOIR (arXiv:2506.07899) uses Mistral-7B as a primary base; EasyEdit special-cases `'mistral'` in editor.py for the ROME/MEMIT/EMMET/AlphaEdit/CORE codepath and ships a Mistral hparams file. **SC-7** PASS — 7B. **SC-8** NOTED-DEFERRED. **SC-9** PASS — **Apache 2.0** with no MAU clause, no naming requirement, and unrestricted derivative redistribution. The cleanest license posture in the field. **ADMIT.**

### Gemma 2 9B
**SC-1** PASS — gated HF repo `google/gemma-2-9b`. **SC-2** PASS — dense `Gemma2ForCausalLM`. **SC-3** PASS — **42 layers** per Gemma 2 paper (arXiv:2408.00118v3) Table 1 and the Google Developers blog architecture dump. **SC-4** PASS — SentencePiece, 256,128 vocab, pinnable via HF revision SHA. **SC-5** PASS with **strong caveat** — the Gemma 2 paper Table 1 explicitly lists "**Pre-norm: yes; Post-norm: yes**" with RMSNorm wrapping both the input and the output of every attention and MLP sub-block. This dual-RMSNorm topology, combined with logit soft-capping (`attn_logit_softcapping=50.0`, `final_logit_softcapping=30.0`), is a real architectural deviation from canonical MEMIT assumptions: the post-MLP RMSNorm scales the FFN output before residual addition, which changes the statistical definition of MEMIT's value vector v. **SC-6** PASS — RLEdit (arXiv:2502.05759) ran MEMIT on Gemma-2-9B successfully, demonstrating that the topology is editable in practice. AlphaEdit Appendix C.6 covers Gemma 1; EAMET ships `gemma-7b-it` hparams. **SC-7** PASS — 9.24B. **SC-8** NOTED-DEFERRED. **SC-9** PASS with conditions — Gemma Terms of Use (https://ai.google.dev/gemma/terms) §3.1 permits public redistribution of "Model Derivatives" (which the .vindex overlay clearly is per definition 1.1(e)) provided four conditions are met: pass-through of the §3.2 use restrictions and the Gemma Prohibited Use Policy as enforceable downstream terms, a copy of the Gemma Terms shipped with each redistribution, prominent change notices on modified files, and the verbatim NOTICE string "Gemma is provided under and subject to the Gemma Terms of Use found at ai.google.dev/gemma/terms." More restrictive than Apache 2.0 but operable. **ADMIT.**

### Gemma 2 27B
SC-1 through SC-5 mirror the 9B entry; **SC-3** is **46 layers** per arXiv:2408.00118 Table 1 and corroborated by Gemma Scope (arXiv:2408.05147). **SC-6** is the deciding row: **no published paper applies MEMIT to Gemma 2 27B specifically**. Precedent exists only for the 9B sibling (RLEdit). Under a strict reading of SC-6 ("this model family OR architecturally-equivalent sibling"), this passes via the 9B sibling; under a strict same-checkpoint reading it fails. SC-7 PASS (27B), SC-9 PASS with the same Gemma ToU conditions. **CONDITIONAL ADMIT — flag SC-6 weakness; if the operator's bar is "exact-checkpoint precedent," REJECT, otherwise ADMIT by sibling extension.**

### Qwen 2.5 7B
**SC-1** PASS. **SC-2** PASS — dense `Qwen2ForCausalLM`. **SC-3** PASS — **28 layers**, sitting exactly on the SC-3 floor. The L15–L25 target band has 11 layers available, which is the bare minimum permitted by MEMIT-COMPAT-PROTO-1. **SC-4** PASS — Qwen2 tiktoken BPE, 151,936 vocab. **SC-5** PASS — Pre-LN + RMSNorm per Qwen2 technical report (arXiv:2407.10671): "RMSNorm and pre-normalization for training stability." **SC-6** PASS (strong) — EAMET (arXiv:2505.11876) reports MEMIT efficacy/generalization on Qwen2.5-7B at 10K–15K edits; SUIT (arXiv:2509.24502), DMLE (arXiv:2604.08284), MEMIT-Merge (arXiv:2502.07322), μKE (arXiv:2504.01196), and EtCon (arXiv:2512.04753) all run MEMIT on Qwen2.5-7B(-Instruct); EasyEdit's `editor.py` special-cases `'qwen' in model_name.lower()` for the MEMIT padding logic. **SC-7** PASS — 7.61B. **SC-8** NOTED-DEFERRED. **SC-9** PASS — Apache 2.0, no MAU restriction. **ADMIT (caveat: SC-3 is at the floor — only 11 layers in the L15–L25 band; if calibration calls for tighter stratification, this becomes uncomfortable).**

### Qwen 2.5 14B
SC-1, SC-2, SC-4, SC-5 mirror 7B. **SC-3** PASS — **48 layers**. **SC-6** PASS (moderate) — no published paper at exactly 14B, but EasyEdit issues #103 and #109 explicitly discuss running ROME/MEMIT on Qwen-14B with `model_parallel: true` for OOM mitigation, and the architecture is identical to 7B. **SC-7** PASS, **SC-9** PASS (Apache 2.0). **ADMIT.**

### Qwen 2.5 32B
SC-1, SC-2, SC-4, SC-5 mirror 7B. **SC-3** PASS — **64 layers**. **SC-6** PASS (weak) — no published 32B-specific MEMIT paper; precedent is by architectural extension via the EasyEdit codepath. **SC-7** PASS, **SC-9** PASS (Apache 2.0; the Qwen blog explicitly calls out that "all open-source models, except for the 3B and 72B variants, are licensed under Apache 2.0"). **ADMIT (caveat: SC-6 is by architectural inheritance only).**

### Qwen 2.5 72B
SC-1, SC-2, SC-4, SC-5 mirror 7B. **SC-3** PASS — **80 layers**. **SC-6** PASS (weak) — same architectural-extension argument as 32B. **SC-7** PASS. **SC-9** PASS with conditions — **custom Qwen License**, not Apache 2.0 (Qwen blog: "All our open-source models, except for the 3B and 72B variants, are licensed under Apache 2.0"). The Qwen License §2 grants modification and derivative-work creation; §3 permits public redistribution under license-bundling and notice-attribution conditions; §4 imposes a >100M MAU commercial threshold; §5(b) requires "Built with Qwen" or "Improved using Qwen" attribution on distributed derivatives. .vindex overlay redistribution is permitted with these conditions. **ADMIT (caveat: license is materially heavier than Apache 2.0 and SC-6 is inferred, not demonstrated).**

### Phi-3 medium (14B)
**SC-1** PASS — `microsoft/Phi-3-medium-4k-instruct`. **SC-2** PASS — "dense decoder-only Transformer model" verbatim from the model card. **SC-3** PASS — **40 layers** per Phi-3 technical report (arXiv:2404.14219): "phi-3-medium … has 40 heads and 40 layers, with embedding dimension 5120." **SC-4** PASS — LLaMA-style SentencePiece, 32,064 vocab. **SC-5** PASS — Pre-LN + RMSNorm; HF docs note "very similar to Llama with the main difference of Phi3SuScaledRotaryEmbedding." **SC-6 FAIL** — EasyEdit's compatibility matrix lists T5/GPT/LLaMA/Baichuan/ChatGLM/InternLM/Qwen/Mistral but not Phi/Phi-3. The closest precedent is phi-1.5 in AlphaEdit Appendix C.6, but a 2025 paper (arXiv:2508.07075) explicitly flags Phi-3-mini as challenging for ROME/MEMIT-style methods: "compact LLMs like Phi-3-mini due to their constrained parameter space ... [ROME/MEMIT] often struggle with complex, conflicting overrides." This is a *negative* precedent rather than supportive. **STOP — REJECT on SC-6.**

### Falcon 7B
**SC-1** PASS. **SC-2** PASS — causal decoder-only. **SC-3** PASS — 32 layers per HF model card. **SC-4** PASS — custom Falcon BPE, 65,024 vocab. **SC-5** PASS-with-concern — Pre-LN with **plain LayerNorm**, **parallel attention/MLP, single layer norm** per HF model card ("Decoder-block: parallel attention/MLP with a single layer norm"). This GPT-J-style topology is supported by SC-5 as long as the architecture flag is set, but it is a deviation from the canonical sequential MLP→residual write path that some MEMIT forks assume. **SC-6 FAIL under strict reading** — Falcon is not in EasyEdit's official compatibility matrix and not in the canonical kmeng01/memit. The single arxiv reference applying MEMIT to Falcon-7B is EAMET (arXiv:2505.11876), which is on the strict edge of "documented MEMIT precedent" (it uses Falcon-7B as a baseline target with reported MEMIT efficacy numbers at 15K edits). Under the **strict posture** of SC-6 ("BOTH canonical precedent AND a modern 2024-2026 release"), Falcon fails the canonical-precedent half: kmeng01/memit does not support it, and EasyEdit's official matrix does not list it. **STOP — REJECT on SC-6 strict posture** (would be a borderline ADMIT under lenient reading via EAMET alone).

### Falcon 40B
SC-1, SC-2, SC-4 mirror Falcon 7B. SC-3 PASS — 60 layers per config.json. SC-5 same parallel-block concern, with two layer norms instead of one. **SC-6 FAIL** — no MEMIT precedent at all; even the EAMET paper that covered Falcon-7B did not cover Falcon-40B. **STOP — REJECT on SC-6.**

### Yi 34B
**SC-1** PASS. **SC-2** PASS — `LlamaForCausalLM` architecture verbatim. **SC-3** PASS — 60 layers per config.json. **SC-4** PASS — LLaMA-derived SentencePiece, 64,000 vocab. **SC-5** PASS — Pre-LN + RMSNorm, identical to LLaMA-2. **SC-6 FAIL** — no published MEMIT paper applies to Yi-34B (or any Yi variant); not in EasyEdit's official compatibility matrix. The architecture is LLaMA-2-compatible, so MEMIT *should* transfer with a hparams adjustment, but the SC-6 strict posture requires demonstrated precedent, not architectural plausibility. **STOP — REJECT on SC-6.** This is the most painful rejection in the field: architecturally Yi 34B is the strongest non-LLaMA MEMIT-suitable target, but the published-precedent gap is real.

### GPT-J 6B (canonical reference)
**SC-1** PASS — ungated `EleutherAI/gpt-j-6b`. **SC-2** PASS — dense decoder-only. **SC-3** PASS — **28 layers** exactly at the floor; HF model card: "28 layers with a model dimension of 4096, and a feedforward dimension of 16384." **SC-4** PASS — GPT-2 BPE, 50,257 vocab. **SC-5** PASS — Pre-LN with plain LayerNorm and **parallel** attention/FFN; deviates from RMSNorm-sequential family but is the original MEMIT target topology. **SC-6** PASS (canonical) — `kmeng01/memit/hparams/MEMIT/EleutherAI_gpt-j-6B.json` ships in the reference repo; the original ICLR 2023 MEMIT paper Table 2 reports numbers on GPT-J at 10,000 edits. AlphaEdit, EAMET, MEMOIR, and dozens of other papers use GPT-J as a primary baseline. **SC-7 FAIL** — 6B is below the 7B SC-7 floor. **SC-8** NOTED-DEFERRED. **SC-9** PASS — Apache 2.0, no restrictions. Operational scoring: SC-7 fails, SC-8 deferred, SC-9 passes — only 1 of 3 operational constraints clearly passes (SC-9), so under a strict reading of "≥2 of 3 must be satisfied" GPT-J is borderline on the operational gate. However, SC-8 is deferred, so SC-9 alone effectively counts as 1-of-2-actively-evaluated, which puts it above the 50% bar. **CONDITIONAL ADMIT as a calibration baseline only** — not recommended as the production target, but invaluable for reproducing canonical numbers before staking the architecture profile on a 2024-era model.

## 4. Short list

**1. LLaMA 3.1 8B (primary recommendation).** SC-6 precedent is the deepest in the field outside GPT-J: Yoon et al.'s direct ROME/MEMIT/EMMET study (arXiv:2405.00664), AlphaEdit's primary experiments (arXiv:2410.02355, ICLR 2025 Outstanding), Gupta et al.'s 10K-edit lifelong-editing study (arXiv:2502.01636), HalluEditBench, and EasyEdit's first-class LLaMA hparams template all converge on this exact checkpoint. SC-7 sits in the **8B sweet spot** — large enough that the L15-L25 target band has 11 deep layers, small enough that MEMIT covariance precomputation completes in ~40 GPU-hours rather than the 36+ hours per layer that 70B variants require. SC-9 license posture is the **lightest non-Apache option**: the "Built with Llama" attribution and "Llama-" name prefix on the .vindex artifact are mechanical compliance, the >700M MAU clause is irrelevant for research, and Meta explicitly grants derivative-work redistribution. Downstream calibration risk is the lowest of any modern candidate: tokenizer is stable across the 3.1/3.3 family, the architecture is the de facto industry baseline that EasyEdit, AlphaEdit, EAMET, RLEdit, and CaKE all target, and the 32-layer depth gives clean L3-L8 ROME edit slots and L10-L20 MEMIT bands matching published configurations.

**2. Mistral 7B v0.3 (de-risked secondary).** SC-6 strength is comparable to LLaMA 3.1 8B — RLEdit (arXiv:2502.05759) explicitly names Mistral-7B-v0.3 as a primary base model alongside Llama-3-8B and Gemma-2-9B; HalluEditBench, MEMOIR, and SCR all report MEMIT numbers on Mistral-7B; EasyEdit special-cases Mistral by name in its editor. SC-7 is at the **7B base of the sweet spot**, slightly smaller than LLaMA 3.1 8B but with the same 32-layer depth, so the L15-L25 band is identical. SC-9 posture is the **strongest in the entire field**: clean Apache 2.0 with no MAU clause, no naming requirement, no Prohibited Use Policy pass-through — this is the candidate where .vindex public redistribution carries zero compliance overhead. Downstream calibration risk is moderately higher than LLaMA 3.1 8B because the v0.3 tokenizer extension (32,000 → 32,768) means EasyEdit's pre-shipped Wikipedia stats are not directly reusable and must be recomputed; budget ~40 GPU-hours for this. Recommended posture: select Mistral 7B v0.3 if the operator anticipates redistributing the .vindex through any channel where LLaMA license attribution is operationally awkward (anonymous artifact registries, non-named research drops).

**3. GPT-J 6B (canonical calibration baseline).** SC-6 strength is **maximal** — this is the original MEMIT paper's primary target and the reference checkpoint in `kmeng01/memit/hparams/MEMIT/EleutherAI_gpt-j-6B.json`; all subsequent papers calibrate against published GPT-J numbers. SC-7 **fails at 6B**, which is why this is recommended only as a calibration-and-reproduction baseline, not as the production target. SC-9 is **Apache 2.0 with zero encumbrances** — the strongest license posture in the field. The downstream calibration value is unique: the operator can reproduce the original ICLR 2023 MEMIT Table 2 numbers (10,000 edits on CounterFact and zsRE) on GPT-J before pinning the architecture profile to a 2024-era model, and any deviation surfaces a toolchain bug rather than a model-specific quirk. The parallel attention/MLP topology (shared with Falcon) means GPT-J does *not* serve as a like-for-like proxy for sequential pre-RMSNorm families like LLaMA or Mistral; treat it as a pipeline-correctness oracle, not a transfer-learning target.

The fourth candidate worth a single mention: **Gemma 2 9B** is the strongest non-LLaMA, non-Mistral admitted candidate, with a clean 42-layer depth (giving generous L15-L25 stratification room) and direct MEMIT precedent via RLEdit. The hybrid pre+post-RMSNorm topology and logit soft-capping are real but tractable architectural deviations — RLEdit demonstrated empirical success at the 9B size. The Gemma Terms of Use are operationally heavier than the LLaMA license (downstream consumers must accept the full ToU and Prohibited Use Policy via pass-through), which is the main reason it sits below LLaMA 3.1 8B and Mistral 7B v0.3 in the ranking.

## 5. Key open questions for the operator

The recommendation above is contingent on the following items, which the operator must resolve before signing the architecture_profile.json:

1. **Production-domain corpus characteristics.** The candidate ranking assumes a roughly LLaMA-3-pretraining-equivalent fact distribution (English-dominated, technical-and-encyclopedic). If the production CFB Domain stratum will include heavy Chinese, Japanese, or Korean fact density, **Qwen 2.5 7B or 14B becomes more competitive** because of its 151,936-vocab Qwen tokenizer's superior CJK coverage; otherwise the 128K LLaMA-3 tokenizer is sufficient. Disclose CFB Domain language stratum sizing.

2. **Strictness of SC-6.** Three candidates (LLaMA 3.1 70B, LLaMA 3.3 70B-Instruct, Gemma 2 27B, Qwen 2.5 32B/72B) admit only via "architecturally-equivalent sibling" reasoning rather than direct same-checkpoint precedent. The protocol's "STRICT POSTURE" wording suggests the operator wants direct precedent; Falcon 7B was REJECTed on this basis even though EAMET (arXiv:2505.11876) reports MEMIT numbers on it. **Operator should rule explicitly: does same-checkpoint precedent matter, or does same-architecture-class suffice?** This rule binds whether Gemma 2 27B and the larger Qwen 2.5 sizes survive review.

3. **License-redistribution channel.** If the .vindex artifact will be redistributed via a channel that cannot enforce click-through accept (e.g., bare GitHub release, anonymous artifact mirror, peer-to-peer share), then Apache-2.0 candidates (Mistral 7B v0.3, Qwen 2.5 7B/14B/32B, GPT-J 6B, Falcon if it had passed SC-6) become strictly preferable to LLaMA Community License or Gemma Terms of Use. **Operator should confirm the redistribution channel.**

4. **Base vs Instruct posture.** The original MEMIT paper localizes facts on **base** models. Editing **Instruct** variants is published practice (HalluEditBench, μKE, EtCon, several others use -Instruct), but causal-tracing locality assumptions can shift slightly. LLaMA 3.3 is **Instruct-only**, which is the principal reason it sits below LLaMA 3.1 70B in the ranking. **Operator should rule whether Instruct-only candidates are acceptable** — this decision binds LLaMA 3.3 70B's status.

5. **GPU memory ceiling for Stage 1 (SC-8).** Currently NOTED-DEFERRED, but the deferral matters for the 70B candidates: CaKE (arXiv:2503.16356) reports MEMIT on Llama-3-70B-Instruct requires 2× A100 80GB minimum and FastMEMIT (arXiv:2506.04226) for tractable covariance precomputation; Sleeping LLM project reports OOM at ~30 facts on 2×H100. The 70B tier may be operationally infeasible on a single-machine setup even if all hard constraints pass.

6. **Tokenizer stability commitment.** SC-4 was scored PASS for all admitted candidates based on HuggingFace revision-SHA pinnability. The operator should confirm that the .vindex deployment will pin tokenizer.json by SHA in the architecture_profile.json (not just by repo name) — particularly relevant for Mistral, where v0.3 vs v0.2 tokenizer differs in vocab size, and for LLaMA 3.3 vs 3.1 where the tokenizer is shared but Meta has historically published silent fixes to tokenizer.json without bumping the model version.

7. **Calibration-to-production transition plan.** If GPT-J 6B is used as the calibration baseline (as recommended in §4), the operator must commit to the migration cost of re-running covariance precomputation, re-tuning K* and L* indices, and re-validating the .vindex schema when transitioning to the production checkpoint (LLaMA 3.1 8B or Mistral 7B v0.3). Budget ~40 GPU-hours for the second precomputation pass plus calibration sweep time. If this migration cost is unacceptable, skip GPT-J calibration and accept the additional toolchain-validation risk.
